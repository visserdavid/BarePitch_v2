Goal:
Build the smallest complete flow that proves BarePitch can run locally from setup through a finished basic match summary. This milestone may use a temporary local developer login only if needed, but it must not be production-enabled.

Implement these slices:
- Phase 0: repository/project foundation.
- Phase 1: database foundation for the core MVP entities needed by the slice.
- Thin setup for club, season, phase, team, user/team role, formation, sample players, and one coach/admin-capable account.
- A minimal team context.
- A minimal player list.
- A minimal match creation flow.
- A minimal match preparation path with formation and lineup persistence.
- A minimal live match screen.
- Start match, register own/opponent regular goals, finish match.
- A basic finished match summary showing final score and timeline.

Required architectural shape:
- Follow the directory and layer rules in the architecture doc.
- Do not put multi-entity writes directly in controllers.
- For write paths, include route/entry point, validator, policy check, service logic, repository access, transaction handling where needed, and tests proportionate to risk.
- Use PHP, MySQL, plain CSS, and vanilla JavaScript only.
- Do not introduce Laravel, Node.js, Docker, Redis, frontend frameworks, or a build pipeline.

Critical behavior to preserve:
- Server is authoritative for match state and score.
- Score must be recalculated from match events, not blindly incremented as source of truth.
- Match states are only documented states: planned, prepared, active, finished.
- A match cannot start unless prepared.
- A finished match remains viewable and must not restart.
- Bench players have no lineup coordinates.
- One active field slot may contain at most one player.

UI requirements:
- Mobile-first layout.
- Stable, calm hierarchy.
- Use bottom navigation only for durable top-level destinations.
- Use contextual in-screen controls for live match actions.
- Critical transitions need confirmation; do not rely on icon-only controls for destructive or state-changing critical actions.

Explicitly do not implement yet:
- Full magic-link email delivery if it blocks the slice.
- Ratings.
- Training.
- Dashboard refinements.
- Livestream.
- Finished-match correction UI.
- Extra time.
- Penalty shootout.
- Advanced statistics.

Acceptance criteria:
- The root URL renders a BarePitch page.
- Database schema installs cleanly.
- Seed data creates one club, season, phase, team, user role, formation, and enough players.
- A user can enter the app in local development and operate in a team context.
- A coach/admin-capable user can create a match.
- A match can be prepared only after valid attendance, formation, and lineup requirements are met.
- A prepared match can start.
- Regular goals can be registered for BarePitch and opponent.
- Score is derived from events and remains correct after refresh.
- Match can be finished and summary remains viewable.
- Protected write routes enforce server-side authorization and CSRF where the infrastructure exists.
- Invalid transitions are rejected safely.
- Tests cover schema install/seed, basic authorization, match preparation requirements, start planned-match rejection, goal score integrity, and finished-match no-restart behavior.

Before finishing:
- Run the available test suite or the closest project-specific verification command.
- Run PHP syntax checks on changed PHP files.
- Document any known gap as a follow-up, not as hidden behavior.
- Do not label this as `v1.0.0`; this milestone is pre-MVP `v0.1.0`.
