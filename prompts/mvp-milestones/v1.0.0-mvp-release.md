Goal:
Finalize the first usable BarePitch MVP release. Do not expand scope. Only fix release-blocking defects, documentation drift, migration issues, test failures, and operational gaps.

MVP release must include:
- Authentication.
- Team context.
- Administrator setup for club, season, phase, team, user, and role assignment.
- Player management.
- Guest players.
- Match creation.
- Match preparation.
- Formation and lineup grid.
- Live match start and regular period flow.
- Goal registration.
- Substitutions.
- Cards and red-card restrictions.
- Penalties during match.
- Extra time.
- Penalty shootout.
- Match finish.
- Basic summary.
- Livestream.
- Finished-match corrections.
- Audit log.
- Locking and concurrency.
- Basic statistics.
- Basic internationalization support.
- Security baseline from the MVP scope and hardening docs.

MVP release may exclude:
- Advanced analytics.
- Heatmaps.
- Tactical whiteboards.
- AI recommendations.
- Push notifications.
- Chat/social features.
- External league integrations.
- Automatic fixture imports.
- Advanced dashboard widgets.
- Parent portals.
- Player photos.
- Advanced media uploads.
- Realtime websocket infrastructure.
- Mobile apps.
- Offline-first architecture.
- Advanced exports.
- Tournament systems.
- Training session management.
- Training attendance workflows.
- Post-match ratings.

Release verification tasks:
- Run a clean schema install and seed.
- Run all automated tests.
- Run PHP syntax checks.
- Perform the documented MVP acceptance scenario.
- Perform the recommended first public test simulation: one coach, one team, one live match, smartphone-sized viewport, real substitutions, cards, goals, livestream, and post-match corrections.
- Verify no manual database intervention is needed for normal use.
- Verify score consistency after refresh and after correction.
- Verify lineup consistency after substitution and red card.
- Verify livestream works, expires, can be stopped, and reflects corrections while active.
- Verify authorization for administrator, coach, trainer, and team manager.
- Verify public token endpoints do not leak unnecessary validity details.
- Verify production-facing error handling does not expose stack traces.
- Verify docs match current behavior.

Release notes to produce:
- Scope.
- Included changes.
- Data or migration impact.
- Security impact.
- Verification summary.
- Known follow-up issues.

Release-blocking criteria:
- A coach cannot complete the full match flow.
- Administrator setup requires direct database edits.
- Score can become inconsistent.
- Lineup can become inconsistent.
- Red-card restrictions fail.
- Finished-match corrections fail or are not audited.
- Livestream fails or leaks token validity details.
- Authorization fails.
- CSRF is missing on MVP write routes.
- Magic login tokens are plaintext, reusable, or non-expiring.
- Public errors expose internals.

Acceptance criteria:
- Complete MVP flow works without developer intervention.
- All included MVP modules function together.
- Permissions work correctly.
- Corrections work correctly.
- Livestream works correctly.
- Basic statistics are correct for MVP data.
- Security baseline is implemented.
- Documentation is aligned.
- Release notes are prepared.
- The release can be tagged as `v1.0.0` only after all release-blocking criteria are cleared.

Before finishing:
- State exactly what was verified.
- State any residual risk or follow-up issue.
- Do not invent a different release version. The MVP release version is `v1.0.0`.
