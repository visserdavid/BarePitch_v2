Goal:
Make planned-to-prepared match preparation reliable, enforceable, and usable on mobile.

Implement these slices:
- Phase 5 match preparation.
- Preparation screen.
- Attendance section.
- Internal and external guest player selection.
- Formation selector.
- 10 row by 11 column lineup grid.
- Bench list with automatic assignment.
- Prepare match action.
- Server-side validation for all preparation requirements.

Required rules:
- A match may become prepared only when at least 11 players are present.
- The maximum player count must not be exceeded.
- A formation must be selected.
- All starting positions must be filled.
- All starters must be present.
- No starter may be injured.
- Only one player may occupy an active field slot.
- Bench players have no coordinates.
- Lineup stores current state only, not historical positional replay.
- Guest status exists in match context.
- Internal guests must come from another team within the same club.
- External guests persist and can be reused.

Authorization requirements:
- Coach and administrator can prepare matches if the authorization matrix allows the action.
- Trainer and team manager cannot perform unauthorized preparation writes unless explicitly allowed by the docs.
- All write routes require CSRF and server-side policy checks.

UI requirements:
- Mobile-first.
- Selection/drag behavior may be implemented with vanilla JavaScript, but the server remains authoritative.
- Grid and bench states must remain clear after validation errors.
- Critical prepare confirmation should require a deliberate action.

Explicitly do not implement yet:
- Full live match expansion beyond start support.
- Substitution flow.
- Cards.
- Penalties.
- Shootout.
- Livestream.
- Corrections.

Acceptance criteria:
- Coach can mark players present, absent, and injured.
- Coach can add internal guest players only from the same club.
- Coach can create/select external guest players.
- Coach can select a formation.
- Coach can fill every starting position.
- Non-starting present players automatically appear on bench.
- Preparing with 10 present players is blocked.
- Preparing with too many players is blocked.
- Preparing with a missing formation is blocked.
- Preparing with an incomplete lineup is blocked.
- Preparing with an injured starter is blocked.
- Valid lineup transitions match from planned to prepared.
- Tests cover data integrity scenarios MP-01 through MP-06, LU-01 through LU-04, and GP-01 through GP-03 from the test scenarios doc.

Before finishing:
- Run available tests and PHP syntax checks.
- Verify no client-side state is trusted as authoritative without server validation.
- Update docs only for discovered gaps or route changes.
