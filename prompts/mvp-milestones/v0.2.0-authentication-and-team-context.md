Goal:
Replace or harden the initial access path so users authenticate through the documented session and magic-link model, then operate inside an authorized team context.

Implement these slices:
- Phase 2 authentication and team context.
- Magic-link request route and callback route as specified.
- One-time expiring magic-login tokens stored hashed.
- Logout.
- Secure session handling.
- Active team selection.
- Current user helper.
- Current team helper.
- Role loading.
- Server-side permission helpers/policies.
- No-role and no-team-access handling.
- Disable any temporary developer login outside local development.

Required behavior:
- Users without role access cannot use the application.
- Multiple team support must work through team context selection.
- Multiple roles per user must be represented according to the schema and authorization docs.
- Protected routes reject unauthenticated users.
- Write routes must authorize server-side; UI hiding is not sufficient.
- Public/auth token failures should avoid leaking token validity details.
- Login tokens must be one-time use and expire.

Security requirements:
- Store magic-link token hashes, never plaintext tokens.
- Use prepared statements for all database access.
- Apply CSRF protection to state-changing routes.
- Configure session cookie flags appropriately for the environment.
- Define idle and absolute session lifetime rules in code or configuration.
- Add basic rate limiting for login token requests if the project already has the rate-limit foundation; otherwise create the minimal foundation required by the docs.

Explicitly do not implement yet:
- Social login.
- Password login.
- Public registration.
- Multi-factor authentication.
- Non-MVP identity features.

Acceptance criteria:
- User can request a magic login link.
- User can authenticate with a valid unexpired token once.
- Reusing the same token fails safely.
- Expired token fails safely.
- User can log out.
- User can select an active team when they have access to more than one.
- User without team role sees a safe no-access state and cannot reach protected routes.
- Coach, trainer, administrator, and team manager roles load from `user_team_role`.
- Server-side policies are used on protected write routes.
- Temporary developer login cannot be active in staging or production configuration.
- Tests cover token hashing, token expiry, token one-time use, no-role access denial, team context switching, unauthenticated protected route rejection, and unauthorized write rejection.

Before finishing:
- Run available tests and PHP syntax checks.
- Update docs only if implementation reveals a documented behavior gap or conflict.
- Do not expand scope beyond authentication and team context.
