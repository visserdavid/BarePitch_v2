Goal:
Make the team setup and planned-match setup flows stable enough for repeated use.

Implement these slices:
- Phase 3 player and team management.
- Phase 4 match creation.
- Administrator setup screens needed for MVP: club, season, phase, team, user, and role assignment if not already complete.
- Player list, create, edit, detail, and deactivate.
- Player season context creation and maintenance.
- External guest player creation.
- Match list, create, edit, and detail.
- Phase assignment, opponent text, home/away, match type, half duration, optional extra-time duration.

Authorization requirements:
- Player management: administrator and team manager can manage; coach and trainer are view-only.
- Match management: coach and administrator can manage; trainer and team manager can view.
- Admin setup routes must follow the authorization matrix and recent-authentication requirements where documented.

Data integrity requirements:
- Player deactivation must not delete historical match data.
- Player season context must be created correctly for team-season usage.
- External guest players persist and are not tied to a team as normal squad players.
- Match belongs to the active team and a valid phase for the team/season context.
- Invalid phase/team combinations are rejected server-side.
- Planned matches can be edited only within documented state constraints.

UI requirements:
- Mobile-first list and form layouts.
- Forms show validation errors near fields.
- Avoid dashboard-like overload.
- Keep navigation stable and predictable.

Explicitly do not implement yet:
- Match preparation beyond links into an existing stub.
- Live match features beyond existing vertical-slice behavior.
- Advanced player statistics.
- Ratings.
- Player photos.
- Training workflows.
- Parent/contact data.

Acceptance criteria:
- Administrator can create minimum setup entities required for a coach to work.
- Team manager can add and edit a player.
- Player appears in team player list with season context.
- Player can be deactivated and disappears from active selection lists without losing historical references.
- External guest player can be created and reused later.
- Coach can view player information but cannot manage players.
- Coach can create and edit a planned match.
- Match appears in match list and detail.
- Trainer can view but not edit match.
- Invalid player data and invalid match data are rejected server-side.
- Tests cover player validation, player authorization, deactivation behavior, season context creation, external guest creation, match creation authorization, invalid phase/team rejection, and trainer view-only access.

Before finishing:
- Run available tests and PHP syntax checks.
- Confirm routes match the route/API specification or document any required spec update.
- Keep changes scoped to setup, players, guests, roles, and planned matches.
